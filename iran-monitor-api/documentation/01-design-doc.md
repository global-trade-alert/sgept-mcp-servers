## Problem Statement

The Iran Conflict Scenario Monitor (`jf-thought/sgept-analytics/iran-monitor/`) is a cron-driven pipeline that produces 4-week-horizon probability assessments for 8 canonical scenarios every 6 hours and publishes the results as a static report. An external institutional buyer (risk desk: hedge fund / sovereign wealth / commodities / reinsurance, "strong intent, not yet paid") has asked for agent-to-agent or human-to-agent access — specifically, the ability to query the perspective stack with **user-defined scenarios outside the canonical 8** and receive a probability assessment with reasoning trace.

The design challenge: transform a fixed-output cron pipeline into a queryable agent that preserves the quality safeguards of the existing system — agent-independence enforcement, source verification, peripheral signal tracking, Bayesian discipline — while operating on demand against novel scenarios.

## Demand Evidence

- Named buyer with concrete workflow ("strong intent, not yet paid")
- Buyer category: institutional risk / finance desk. Promoted for non-consensus calls that pay off; fired for missing fat-tail moves.
- Query shape (verbatim user framing): "What's the probability that Iran will launch a cyber attack on Germany?" — assessed through the 14 perspective agents using accumulated evidence (since Feb 2026), longitudinal reasoning traces, and an immediate scenario-targeted gathering pass for fresh sources.
- Wedge price estimated at £10–20k/month per desk. Three paying desks ≈ runway for one dedicated engineer.

## Status Quo

Risk desks currently get geopolitical scenario assessments from:

- Subscription analyst houses (Eurasia Group, RANE, Stratfor descendants) — PDF briefings, weekly cadence, single house view, not machine-callable.
- Internal analysts — slow, expensive, inconsistent methodology.
- Crowd-sourced forecasting (Metaculus, Good Judgment Open) — not real-time enough for buyer-grade decisions.
- Generic LLM prompting — no verified evidence base, no auditable reasoning, no defensible methodology.

The cobbled-together status quo: PDFs + Bloomberg terminal + analyst Slack + occasional Eurasia call. The actionable gap: nothing on the market combines machine-callable + verified-source corpus + multi-perspective agentic assessment + audit trail.

## Target User & Narrowest Wedge

Institutional risk desk at a hedge fund, sovereign wealth fund, commodities house, or reinsurer. The actual human: a senior risk analyst or portfolio risk officer whose desk holds Iran-exposed positions (energy, MENA equities, sovereign debt, shipping, gold). Their KPI: avoiding fat-tail surprise; their decision cadence: daily to intraday.

**Narrowest wedge:** Async novel-scenario endpoint. Risk desk POSTs a scenario description + horizon + tier; we queue; the perspective stack runs against latest verified intelligence; structured assessment returned via polling within the latency budget defined in Constraints (canonical reference: Constraints section).

## Premises (load-bearing assumptions for the design)

1. **Tiered GATHER.** Standard tier binds to the latest cron intelligence-base hash. Premium tier runs a scenario-targeted live GATHER pass before ASSESS. Both share an identical ASSESS pipeline and identical safeguards. Tier is a query parameter; pricing differs. _(Codex challenged this; user maintained on commercial grounds — pilot data will validate or falsify within the first 90 days.)_
2. **Buyer is institutional risk desks at £10–20k/month/seat.** Three paying desks ≈ a dedicated engineer.
3. **The moat is the intelligence base + multi-perspective stack + Bayesian discipline, NOT the assessment.** Raw probability answers are commodity LLM output; verified-source corpus, longitudinal reasoning history, and signed audit trail are not.
4. **Quality safeguards are mechanical, not aspirational.** Agent-independence enforced via prompt isolation. Aggregation method fixed per query (no α-transition mid-query). Per-query audit record signed against intelligence-base hash. Reasoning trace returned to caller.
5. **Public website stays — repurposed as top-of-funnel marketing.** Free static report seeds inbound demand for the paid queryable API.
6. **Calibration on novel scenarios is impossible.** No ground truth. Quality signal becomes reasoning-trace integrity + inter-perspective divergence flagging. The 8 canonical scenarios remain Brier-tracked.
7. **Out of scope for wedge:** counterfactuals ("if X had happened"), synchronous chat surfaces, multi-conflict expansion (Ukraine / Taiwan / etc.), free read-only tier.
8. **Procurement readiness is a Phase-2 deliverable.** Phase 1 ships a single-page data-handling note negotiated with the named pilot only. Full procurement pack (DPA, model card, sanctions disclosure) lands in Phase 2 once the wedge is validated. _Reasoning: building the procurement pack against a hypothetical generic buyer is premature; build it against the actual pilot's procurement questionnaire._
9. **Actionability is a Phase-2 deliverable.** Phase 1 uses polling only; threshold semantics and webhook callbacks land in Phase 2. _Reasoning: Phase 1's job is to learn what threshold definitions buyers actually want by capturing pilot feedback — building webhooks before that risks shipping the wrong action surface._

## Approaches Considered

### Approach A: HTTP REST + Async Worker (recommended; minimal viable)
HTTP `POST /v1/queries` + `GET /v1/queries/{id}` only. SQLite-backed queue (no Redis). Single Python worker (uv-managed), concurrency = 1. Reuses existing perspective agents as Claude subagents with same independence enforcement as cron. Standard binds to latest cron base hash; Premium runs scenario-targeted GATHER pass first. Polling only (no webhooks). MCP stdio wrapper deferred to Phase 2 once the pilot confirms the agent-to-agent vs human-to-agent split. Effort: M–L (~5–7 weeks, includes 3–5 day subagent-invocation spike up front). Risk: Med.

### Approach B: Inference Platform (long-term shape)
Same surface as A plus: worker pool with tier-aware scheduling, Postgres-backed query history, programmable thresholds + webhook callbacks, thin web chat UI for human analysts, procurement pack (DPA, model card, data-handling, sanctions). Effort: M–L (~6–8 weeks). Risk: Med.

### Approach C: Productize the Stack as a Library (lateral, kept on the shelf)
Ship a Python package + signed/versioned intelligence base. Buyer runs ASSESS on their own compute with their own LLM keys. Revenue from data licensing. Trades the moat (open-sourced perspective prompts) for procurement-friendliness. Effort: M (~4–5 weeks). Risk: Med-High.

## Recommended Approach

**Approach A** for the wedge build, with a planned pivot to Approach B once the first paying pilot signs (i.e. once procurement readiness becomes a payment-blocker rather than a hypothetical). Rationale:

- Ships in 7–10 weeks against the existing iran-monitor codebase (revised twice during adversarial review from an initial 2–3 weeks).
- Validates the wedge with the named buyer before committing platform scaffolding.
- Preserves all current quality safeguards (independence enforcement, source verification, intelligence-base hash, reasoning trace).
- Codex's stress test (>70% Premium use, or >10pp Standard/Premium deltas) is detectable from query logs within the first 30–90 days of pilot use — A defers the decision until there's evidence.
- B's webhook + procurement deliverables become the natural Phase 2 once the wedge is validated.

### Architecture — what the worker actually does on a query

**Information assets consulted per query (Standard and Premium):**

| Asset | Used how |
|---|---|
| `data/probabilities*.csv` | Bayesian prior on canonical 8 (or adjacent scenarios). For novel scenarios, Tetlock-forecaster synthesizes a cold-start prior from base rates. |
| `data/perspective-assessments/YYMMDD-HHMM/*.json` (~1,260 files) | Per-agent longitudinal reasoning history. Each agent reads its own prior outputs to ground continuity on adjacent topics. |
| `data/briefings/*.md` | Methodological constitution (scenario definitions, Bayesian framework, event elasticity). |
| `data/tracking/peripheral-watch-list.jsonl` | Cross-cycle weak signals. **Primary lookup for novel-scenario queries** — peripheral signals are where non-canonical scenarios first appeared. |
| Latest cron cycle's verified `raw-intelligence.md` | Anchor evidence ("verified state of the world as of T"). |
| `.claude/skills/knowledge/*` | 18+ framework knowledge skills loaded on demand by perspective agents. |
| `legacy/advanced-iran-monitor.md` | Cold-start routing guide for novel-scenario relevance per perspective. |

**Premium-tier addition — scenario-targeted GATHER module (the technically novel piece):**

- Inputs: scenario description, query ID.
- Step 1: LLM-driven keyword/entity extraction from the scenario description (e.g. cyber-Germany → IRGC-CEC, BfV, BSI, Germany-Iran ties, recent Iran cyber ops against Western targets).
- Step 2: Bounded `web_search` pass (n ≤ 20 queries, ≤ 60 fetches). Source verification by URL + publication date — same verification depth as the cron's GATHER phase.
- Step 3: Append to a *query-scoped intel delta* JSONL file at `data/query-deltas/{query_id}.jsonl`. One JSON object per line, schema: `{event, source_url, publication_date, fetched_at, verification_status, hash}`. Dedup against the canonical intelligence base by `source_url`.
- Step 4: Hand off to ASSESS with (canonical base hash + query delta hash) as a unified evidence pack.
- Concurrency: under Phase 1 worker concurrency = 1, only one query runs at a time. Two queries on overlapping scenarios queue serially; their intel deltas are independent files. (Concurrency > 1 is a Phase 2 concern; the per-query delta directory anticipates it.)
- Latency: see canonical commitments in Constraints. Beyond the p99 ceiling, the query times out with a partial audit record marking which perspectives completed.

**ASSESS pipeline (Standard and Premium, identical):**

- Routing layer: an LLM classifier returns the subset of 14 perspective agents relevant for this scenario (frozen default for Phase 1 launch: all 14 minus `solingen-proliferation` and `narang-nuclear` unless the scenario keywords trigger them; buyer may override via `perspectives?` query param).
- Subagent invocation: each perspective agent is invoked as a headless `claude -p` subprocess from the Python worker, with its own working directory, scenario + intelligence-base path + the agent's own historical reasoning files injected via prompt. Agent N receives nothing from agent N-1 (isolation by process boundary, not by prompt convention). **This invocation pattern is the single biggest implementation risk and is the subject of the 3–5 day spike up front.**
- Aggregation: weighted average across selected perspectives, divergence flagged at >15pp (matching cron behavior). No α-transition mid-query.
- Cold-start prior derivation: explicit pipeline step. Tetlock-forecaster runs first with the scenario + base-rate inputs only and produces a prior. All other perspectives then see (scenario + prior + intelligence base + their own history) but NOT each other. Tetlock's prior is part of the audit record.
- Reasoning trace vs audit record: the reasoning trace is the *content* (per-perspective assessments, evidence citations, divergence narrative) — exposed via the `result.perspectives` field. The audit record is the *signed envelope* covering structural metadata + intelligence-base hash + result summary — exposed as `audit_record` with detached `audit_signature`. Per-perspective reasoning text is stored alongside in `data/query-outputs/{query_id}/perspectives/*.json` (retained 24 months per data-retention policy).
- Worker crash recovery: queries in `running` state at worker restart are re-queued (idempotency: each subagent invocation writes its output atomically by `mv` to its perspective output file; on restart the worker scans for completed perspective outputs and resumes from where it left off). Idempotency depends on the spike validating that re-invoking a perspective agent produces functionally-equivalent output — flagged for spike testing.
- Signing-key compromise procedure: emergency rotation announced via `/.well-known/signing-key-revocation.json` listing revoked key fingerprints; new key issued same day; audit records signed by revoked keys remain verifiable but flagged as "key-revoked-after" in the verification endpoint.

**Surface (Phase 1):**

- HTTP REST only: `POST /v1/queries` and `GET /v1/queries/{id}`. API key auth via `Authorization: Bearer <key>`.
- Per-org rate limit: 30 req/hour Standard, 10 req/hour Premium. Enforced via SQLite-backed sliding-window counter (rebuilt from query log on worker restart — idempotent). 429 with `Retry-After` header on overflow.
- Polling only. MCP stdio wrapper and webhooks deferred to Phase 2.
- Input validation: scenario description ≤ 1,500 chars (stripped of HTML/markdown control chars before reaching agents); horizon ∈ {7d, 14d, 30d, 60d, 90d}; tier ∈ {standard, premium}. Prompt-injection screening: scenario text sandwiched between explicit `START_USER_SCENARIO` / `END_USER_SCENARIO` markers in every perspective-agent prompt; agent system prompts instruct to treat enclosed text as untrusted; additional defense from agent-process isolation (subagents run with no tool access beyond read of intelligence base).
- Error taxonomy: 400 malformed input; 401 missing/invalid API key; 403 valid key, insufficient tier; 404 unknown `query_id`; 410 query archived beyond retention; 429 rate limit; 503 service unavailable (worker down) or partial-quorum result; 504 query exceeded p99 ceiling.

**API response shapes:**

`POST /v1/queries` → 202 Accepted:
```json
{ "query_id": "uuid", "status": "queued", "tier": "standard|premium", "submitted_at_utc": "ISO8601", "estimated_completion_utc": "ISO8601" }
```

`GET /v1/queries/{id}` (status `queued` or `running`):
```json
{ "query_id": "uuid", "status": "queued|running", "perspectives_total": 12, "perspectives_completed": 4, "elapsed_seconds": 240 }
```

`GET /v1/queries/{id}` (status `completed`):
```json
{
  "query_id": "uuid",
  "status": "completed",
  "result": {
    "p_point": 0.18,
    "p_interval": [0.10, 0.28],
    "divergence_flag": false,
    "consensus_summary": "string",
    "perspectives": [ { "name": "tetlock-forecaster", "p_point": 0.14, "key_reasoning": "...", "evidence_urls": [...] }, ... ]
  },
  "audit_record": { /* full schema below */ },
  "audit_signature": "base64-Ed25519"
}
```

`GET /v1/queries/{id}` (status `partial` — quorum met but some perspectives failed) → 200 with body identical to `completed` plus a `partial: true` flag and `failed_perspectives: [...]` listing.

`GET /v1/queries/{id}` (status `failed` — quorum not met) → 503 with partial audit record naming `failed_perspectives`, but no `result.p_point`.

**Quorum rule:** require ≥ ⌈2N/3⌉ successful perspectives for a `completed` verdict where N = perspectives selected by the routing layer (N=12 default → 8 required); 1–N−1 short of quorum → `failed`; quorum met with some failures → `partial`.

**Audit record schema (signed):**

```json
{
  "query_id": "uuid",
  "scenario_text": "string",
  "horizon_days": 30,
  "tier": "standard|premium",
  "intelligence_base_hash": "sha256:...",
  "query_delta_hash": "sha256:... | null",
  "perspectives_invoked": ["tetlock-forecaster", "schelling-bargaining", ...],
  "perspectives_completed": ["tetlock-forecaster", ...],
  "aggregation_method": "weighted_average_v1",
  "result_summary": { "p_point": 0.18, "p_interval": [0.10, 0.28], "divergence_flag": false },
  "evidence_urls": ["https://...", ...],
  "started_at_utc": "ISO8601",
  "runtime_seconds": 1432,
  "version": "1.0"
}
```

- Signature: detached Ed25519 over the canonicalized JSON. Signing key generated on Metis, stored in SOPS-encrypted env. Public key published at `https://api.iran-monitor.sgept.org/.well-known/iran-monitor-signing-key.pub`. Buyer-side verification: standard libsodium-compatible.
- Signing-key rotation: annually; rotation announced 30 days in advance via a new key published alongside the old at `/.well-known/iran-monitor-signing-key.<version>.pub`. Old key remains valid for verification indefinitely.
- API-key rotation: quarterly; per-org. Revocation immediate; rotation grace window 7 days (both old + new accepted).

**Intelligence-base hash — precise definition:**

- SHA-256 over a sorted manifest of `(relative_path, content_sha256)` pairs covering: `data/probabilities.csv`, `data/probabilities_v.csv`, `data/perspective-assessments/` (recursive), `data/tracking/peripheral-watch-list.jsonl`, `data/tracking/aggregation-state.json`, and the latest cycle's `raw-intelligence.md` and `briefing-*.md`.
- Sealed at the end of `run-cycle-cron.sh` Phase 6 (COMMIT) — after git commit + push, the script writes `data/.intel-base-hash` as the canonical hash of the just-committed state. A Standard query reads this file at query start; if the cron is mid-cycle, the file points to the previous fully-committed cycle.
- Concurrency safety: the cron writes `.intel-base-hash.new` then `mv` to `.intel-base-hash` (atomic on POSIX). Workers never read a half-written file.

**Hosting:**

- Self-hosted on existing Metis (Hetzner CPX32). Caddy for TLS termination (existing pattern from mavka). uv-managed Python worker + HTTP server as a single systemd service. SQLite-backed queue + query log + audit (Postgres deferred to Phase 2).
- Subdomain: `api.iran-monitor.sgept.org`. Cloudflare WAF deferred to Phase 2 — Caddy + API-key auth + per-org rate limit is sufficient for the first 3 pilots.

## Open Questions

- **Subagent invocation pattern (load-bearing for the 5–7 week estimate).** The spike validates: does `claude -p` launched from a Python worker, with custom CWD + prompt-injected intelligence-base paths, deliver the same isolation + output quality as the existing cron `/update` flow? If not, fallback is the Claude Agent SDK or a thinner LangGraph orchestration. Spike outputs decide.
- **Cold-start prior derivation:** Tetlock-forecaster alone, or co-stepped with Bayesian-multi-scenario-update knowledge skill? Pilot will reveal.
- **Standard tier intelligence-base freshness:** if the cron is at T-5h when a Standard query arrives, do we trigger an out-of-band cron refresh? Premise #1 says no; pilot data settles this within 90 days.
- **Pilot's procurement questionnaire shape.** The single-page data-handling note for Phase 1 has to be reactive to the first pilot's specific questions — schedule the call before drafting.
- **MCP wrapper priority for Phase 2.** If the named buyer is an institutional risk desk, HTTP REST is the primary surface (curl + LLM tool-calling). If they explicitly want their MCP-native agent to call ours, MCP stdio jumps the queue.

## Success Criteria

- **Spike gate (week 1):** subagent-invocation spike produces ≥3 perspective-agent assessments of a sample novel scenario with isolation verified (output of agent N is independent of agent N-1 — checkable by varying invocation order). Pass = build Phase 1. Fail = redesign invocation pattern before committing.
- First paying pilot signed within 90 days of API launch.
- Per-query latency: matches canonical commitments in Constraints (Standard p50 ≤ 8 min / p99 ≤ 30 min; Premium p50 ≤ 25 min / p99 ≤ 60 min).
- Per-query compute cost: median < $10 Standard, < $25 Premium.
- Reasoning-trace integrity check passes on 100% of queries (intelligence-base hash matches; all cited URLs resolved; per-perspective outputs structurally complete; signature verifies).
- Inter-perspective divergence flag (>15pp) fires on at least one query in the first 50 (evidence the agents are reasoning independently, not collapsing to consensus).
- At least one Codex-style stress test runs by day 90: % of queries selecting Premium, distribution of Standard-vs-Premium probability deltas. If >70% Premium and median delta >10pp, drop Standard tier.
